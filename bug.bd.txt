# Forensic Audit Report: duha-playwright

## 1. Executive Summary

This codebase is a school portal automation suite with 41 backend issues, 32 frontend issues, and 20+ test/config issues discovered. The system is functional but carries **critical security vulnerabilities**, **systemic code duplication**, and **significant test coverage gaps**.

**Top 5 Systemic Failures:**

1. **CRITICAL: Hardcoded session cookies committed to repository** — `leave/fetch_leaves.js` contains full Laravel session cookies and XSRF tokens in plaintext, granting full API access to the school portal.
2. **HIGH: Unsanitized `.env` write endpoint** — `PUT /api/config` writes arbitrary user input to `.env` with no validation, and interpolates keys into `RegExp` without escaping (ReDoS risk).
3. **HIGH: 30%+ code duplication** — WhatsApp logic, normalize functions, month constants, and staff matching are duplicated across 3-5 files each, creating divergent behavior risks.
4. **HIGH: Zero test coverage on critical paths** — 16+ utility files, all extractors, the main orchestrator, and the server have no tests. The `attendanceToPayroll()` bridge function (the payroll integration) is completely untested.
5. **MEDIUM: No error boundaries or error state handling in frontend** — React app crashes on any render error; API failures silently show stale data.

**Business Impact:** The hardcoded credentials represent an active security breach. The code duplication means bug fixes must be applied in 3+ places. The test gaps mean regressions are undetectable.

## 3. Deep Root Cause Insights

### Root Cause 1: Hardcoded Credentials (SEC-01, SEC-02)

**5 Whys:**
1. Why are cookies in the repo? → Developer copy-pasted browser headers for quick API testing
2. Why weren't they removed? → No code review process caught it; no git pre-commit hook for secrets
3. Why no secrets scanning? → `.gitignore` covers `.env` but not JS files with inline secrets
4. Why PII in leave.py? → Data was hardcoded for the initial implementation instead of fetched from API
5. Why not cleaned up? → `leave/` directory is treated as a standalone utility, not part of the main security perimeter

**Blast Radius:** Full API access to the school portal. Employee names, health information (leave reasons), and contact numbers exposed in version control history.

### Root Cause 2: Systemic Code Duplication (SMELL-01 through SMELL-06)

**5 Whys:**
1. Why is WhatsApp logic duplicated? → Server route and reporter were built independently
2. Why not shared? → No extraction of common utilities was planned during initial implementation
3. Why does it keep growing? → Each new feature (server route, reporter, web page) re-implements the same functions
4. Why no consolidation? → No architectural review or refactoring sprints
5. Why is manualMap in 3 places? → Payroll was integrated from 3 different entry points without a shared module

**Blast Radius:** Bug fixes must be applied in 3-5 places. Divergent behavior between `parseDueFromCell` implementations in `duesFilter.ts` (regex `gi` flag, returns max) vs `whatsappReporter.ts` (regex `i` flag, returns first match).

### Root Cause 3: Test Coverage Gaps (TEST-01 through TEST-04)

**5 Whys:**
1. Why are extractors untested? → They depend on Playwright browser context, considered hard to unit test
2. Why no mocks? → Test infrastructure was set up for pure functions only
3. Why does `attendanceToPayroll()` have no tests? → It does filesystem I/O, considered integration-level
4. Why no integration tests? → No CI pipeline runs integration tests against a real or mocked portal
5. Why does `payrollCompute.test.ts` skip in CI? → It `require()`s a file from `js-agv8/` that doesn't exist in CI

**Blast Radius:** Regressions in the payroll bridge, diff engine, spreadsheet writer, and API flattener are completely undetectable.

### Root Cause 4: Frontend Architecture Lapses (FE-01 through FE-09)

**5 Whys:**
1. Why no error boundaries? → React best practices weren't followed during initial build
2. Why no URL routing? → SPA was built as a simple tab-switcher, never upgraded
3. Why no error states? → Development assumed API always succeeds
4. Why pervasive `any` types? → TypeScript was used loosely; no strict mode in web/tsconfig
5. Why Tailwind CDN in production? → Copied from Tailwind docs without removing the dev script

**Blast Radius:** App crashes on any render error. Users cannot bookmark or share links. API failures produce silent blank screens.

### Fix SEC-03: Secure `PUT /api/config`

```typescript
// src/server/routes/config.ts — PUT handler
const ALLOWED_KEYS = new Set([
  'PORTAL_BASE_URL', 'PORTAL_USERNAME', 'PORTAL_PASSWORD',
  'PORTAL_YEAR', 'PORTAL_SHIFT', 'PORTAL_CLASS',
  'PORTAL_DUE_STUDENTS_ONLY', 'PORTAL_MIN_DUE', 'PORTAL_COLUMNS',
  'PORTAL_PERIOD_MONTHS', 'PORTAL_COLUMN_FILTER',
  'EXTRACT_ATTENDANCE', 'EXTRACT_ACCOUNTS_RECEIVABLE',
  'GENERATE_WHATSAPP_DASHBOARD', 'WEB_PORT', 'HEADED',
  'ENABLE_TELEGRAM_NOTIFICATIONS', 'TELEGRAM_BOT_TOKEN', 'TELEGRAM_CHAT_ID',
]);

function escapeRegex(s: string): string {
  return s.replace(/[.*+?^${}()|[\]\\]/g, '\\$&');
}

configRouter.put('/', (req, res) => {
  const updates = req.body as Record<string, string>;
  let content = fs.existsSync(ENV_PATH) ? fs.readFileSync(ENV_PATH, 'utf-8') : '';

  for (const [key, val] of Object.entries(updates)) {
    if (!ALLOWED_KEYS.has(key)) continue;
    if (key === 'PORTAL_PASSWORD' && val === '••••••••') continue;
    const regex = new RegExp(`^${escapeRegex(key)}=.*$`, 'm');
    const line = `${key}="${val}"`;
    content = regex.test(content) ? content.replace(regex, line) : `${content}\n${line}`;
  }

  fs.writeFileSync(ENV_PATH, content, { mode: 0o600 });
  res.json({ success: true });
});
```

### Fix SEC-04: Safe `GET /api/config`

```typescript
const SAFE_KEYS = new Set([
  'PORTAL_BASE_URL', 'PORTAL_YEAR', 'PORTAL_SHIFT', 'PORTAL_CLASS',
  'PORTAL_DUE_STUDENTS_ONLY', 'PORTAL_MIN_DUE', 'PORTAL_COLUMNS',
  'PORTAL_PERIOD_MONTHS', 'EXTRACT_ATTENDANCE', 'EXTRACT_ACCOUNTS_RECEIVABLE',
  'GENERATE_WHATSAPP_DASHBOARD', 'WEB_PORT', 'HEADED',
]);

configRouter.get('/', (_req, res) => {
  const content = fs.existsSync(ENV_PATH) ? fs.readFileSync(ENV_PATH, 'utf-8') : '';
  const vars: Record<string, string> = {};
  for (const line of content.split('\n')) {
    const trimmed = line.trim();
    if (!trimmed || trimmed.startsWith('#')) continue;
    const eqIdx = trimmed.indexOf('=');
    if (eqIdx > 0) {
      const key = trimmed.slice(0, eqIdx);
      if (SAFE_KEYS.has(key)) {
        vars[key] = trimmed.slice(eqIdx + 1).replace(/^["']|["']$/g, '');
      }
    }
  }
  res.json(vars);
});
```

### Metrics to Track

| Metric | Threshold | Action |
|--------|-----------|--------|
| Extraction duration | > 10 min | Alert — possible portal slowness or stuck combo |
| Failed combos per run | > 0 | Log + dashboard indicator |
| API error rate (server) | > 5% in 5 min | Alert — portal may be down |
| File output size | 0 bytes after successful run | Alert — extraction returned empty |
| SSE client count | > 10 | Warn — potential memory pressure |
| SQLite DB size | > 100MB | Trigger archival/cleanup |
| `.env` file modification | Any change | Log audit trail with timestamp |

### SLI/SLO Definitions

| SLI | SLO |
|-----|-----|
| Extraction success rate | ≥ 95% of runs complete without fatal error |
| API response time (p95) | ≤ 500ms for `/api/dashboard`, `/api/dues` |
| Data freshness | Output files ≤ 24h old |
| Web UI availability | ≥ 99% during business hours |

### Alerting Rules

1. **P1 (Immediate):** Hardcoded credentials detected by `gitleaks` → block merge + rotate secrets
2. **P1 (Immediate):** `PUT /api/config` called with non-allowlisted key → log + alert
3. **P2 (Same day):** Extraction failure rate > 10% in 24h → investigate portal changes
4. **P3 (Weekly):** Test coverage drops below threshold → block merge
